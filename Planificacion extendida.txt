eTeam Manager — Planificación extendida del proyecto

Fecha: 11 de marzo de 2026

======================================================================

OBJETIVO DEL DOCUMENTO

Este documento amplía la planificación original del proyecto eTeam Manager
con una estructura más clara, más práctica y mejor dividida por sprints,
actividades y tareas concretas.

La idea no es solo decir "qué módulo toca", sino dejar definido:

- qué se trabaja en cada sprint
- qué actividades lo componen
- qué ejercicios o tareas concretas hay que realizar
- qué resultado debe salir de cada bloque
- cómo saber si ese sprint está realmente terminado

Este plan está pensado para un entorno académico y para una V1 centrada en:

- frontend funcional
- navegación real
- datos simulados (mocks)
- arquitectura preparada para backend y base de datos futura

======================================================================

RESUMEN GENERAL DEL PROYECTO

Nombre del proyecto:
eTeam Manager

Descripción breve:
Aplicación web para la gestión interna de organizaciones y equipos de eSports.
Centraliza en una sola plataforma herramientas como calendario, scrims,
tableros Kanban, notas, roles y organización de miembros.

Objetivo general de la V1:
Construir un prototipo visual completo, estable y coherente que simule el
comportamiento de una aplicación real, sin depender todavía de backend ni
persistencia real.

Duración total estimada:
8 semanas

Carga estimada por semana:
12 horas en total

Carga total aproximada:
96 horas

Metodología:
SCRUM adaptado a proyecto académico

======================================================================

BASE TÉCNICA Y CRITERIOS DEL PROYECTO

El plan extendido se apoya en los documentos base del proyecto y en el estado
actual del repositorio.

Puntos técnicos que condicionan la planificación:

- El proyecto está orientado a frontend modular.
- El stack del repositorio gira en torno a PHP, CSS y JavaScript.
- La interfaz debe mantener una identidad visual competitiva y oscura.
- Los estilos globales deben centralizarse en css/main.css.
- La lógica visual e interacciones deben centralizarse en js/main.js y módulos
  específicos cuando haga falta.
- La estructura debe quedar preparada para conectarse más adelante a backend,
  autenticación real y base de datos relacional.
- El sistema conceptual del producto gira alrededor de usuarios,
  organizaciones, equipos, eventos, scrims, tareas y notas.

Reglas de alcance para esta V1:

- No se desarrolla backend funcional completo.
- No hay persistencia real en base de datos durante esta fase.
- La autenticación es simulada.
- Los datos de negocio se representarán mediante mocks.
- El foco principal es estructura, experiencia de uso y coherencia modular.

ESTRUCTURA SCRUM PROPUESTA

Duración de cada sprint:
1 semana

Ritmo recomendado por sprint:

1) Sprint Planning
   - Revisar objetivo de la semana.
   - Confirmar pantallas, componentes y datos mock necesarios.
   - Identificar dependencias con sprints anteriores.

2) Diseño y preparación
   - Revisar layout, navegación y componentes reutilizables.
   - Preparar estructuras base antes de programar el módulo.

3) Desarrollo
   - Construcción visual del módulo.
   - Integración con mocks.
   - Interacciones y validaciones front.

4) Revisión
   - Comprobar navegación.
   - Verificar consistencia visual.
   - Revisar responsive y estados vacíos.

5) Cierre del sprint
   - Dejar entregable demostrable.
   - Anotar mejoras o bloqueos para el sprint siguiente.

Definición de "terminado" para cualquier sprint:

- La UI del módulo existe y se puede navegar.
- Los componentes principales funcionan a nivel visual.
- Los datos mock están conectados a la interfaz.
- El módulo respeta el estilo global del proyecto.
- No rompe la navegación de los módulos anteriores.

SPRINT 1 — BASE VISUAL, LAYOUT Y AUTENTICACIÓN SIMULADA

Objetivo del sprint:
Construir la base visual de la aplicación para que el proyecto tenga una
estructura reutilizable, una navegación clara y un acceso simulado mediante
login/registro.

Duración estimada:
12 horas

Entregable principal:
Prototipo navegable con layout base compartido, formularios de acceso y sesión
simulada.

ACTIVIDAD 1 — Preparar estructura general de la aplicación (2h)

Qué se tiene que hacer:
- Revisar la estructura actual del repositorio.
- Definir qué partes serán reutilizables entre páginas.
- Separar claramente layout, contenido principal y recursos compartidos.
- Decidir la organización base entre index, includes, páginas y scripts.

Ejercicios o tareas concretas:
- Identificar encabezado, menú, contenedor principal y pie común.
- Crear una estructura repetible para futuras páginas.
- Dejar claro dónde se cargarán CSS globales y JS globales.

Resultado esperado:
- Base estructural común para todas las vistas.

ACTIVIDAD 2 — Diseñar layout compartido y navegación principal (3h)

Qué se tiene que hacer:
- Implementar header, sidebar o navegación superior según la estructura elegida.
- Definir enlaces claros a los módulos principales.
- Garantizar que la navegación sea reutilizable sin duplicar código.

Ejercicios o tareas concretas:
- Crear menú principal con acceso a dashboard, equipos, scrims, calendario,
  boards, notas y ajustes.
- Marcar visualmente el apartado activo.
- Preparar estados de navegación para usuario autenticado y visitante.

Resultado esperado:
- Navegación coherente y lista para escalar.

ACTIVIDAD 3 — Consolidar identidad visual global (3h)

Qué se tiene que hacer:
- Aplicar la paleta de colores oficial.
- Aplicar la tipografía del sistema.
- Crear componentes básicos reutilizables.

Ejercicios o tareas concretas:
- Configurar variables globales de color y tipografía.
- Estilizar botones primarios y secundarios.
- Estilizar inputs, labels, cards, badges y mensajes de estado.
- Revisar márgenes, espaciados y jerarquía visual.

Resultado esperado:
- Sistema visual unificado y consistente desde el primer sprint.

ACTIVIDAD 4 — Construir login y registro (2.5h)

Qué se tiene que hacer:
- Diseñar las vistas o formularios de login y registro.
- Preparar una experiencia visual clara para acceso de usuario.

Ejercicios o tareas concretas:
- Crear campos de email, usuario y contraseña.
- Mostrar mensajes de error y ayuda contextual.
- Añadir CTA claros para cambiar entre login y registro.

Resultado esperado:
- Pantallas de acceso completas a nivel visual.

ACTIVIDAD 5 — Validaciones front y sesión simulada (1.5h)

Qué se tiene que hacer:
- Validar los formularios en frontend.
- Simular inicio de sesión y cierre de sesión.
- Restringir el acceso visual a páginas internas si no hay sesión simulada.

Ejercicios o tareas concretas:
- Validar campos vacíos.
- Validar formato de email.
- Validar longitud mínima de contraseña.
- Guardar sesión en localStorage o sessionStorage.
- Redirigir a login si se intenta entrar sin sesión.

Resultado esperado:
- Flujo de acceso usable y protegido a nivel visual.

Criterios de cierre del sprint:
- El usuario puede entrar y salir de la app de forma simulada.
- Existe un layout común reutilizable.
- La app tiene una base visual sólida y consistente.

- Base clara del módulo antes de maquetarlo.

Qué se tiene que hacer:
- Construir el módulo principal de notas.

Ejercicios o tareas concretas:
- Crear, editar y eliminar notas.
- Añadir campos título y contenido.
- Mostrar notas en formato lista o cards.

Resultado esperado:
- Repositorio de notas funcional a nivel visual.

Qué se tiene que hacer:
- Facilitar clasificación de contenido táctico.

Ejercicios o tareas concretas:

- Filtrar por tag.
- Mostrar claramente qué filtros están activos.

Resultado esperado:
- Módulo de notas más ordenado y útil.

- Revisar el sistema completo antes de cerrar la V1.

Ejercicios o tareas concretas:

Criterios de cierre del sprint:
- Todos los módulos principales están presentes.
- La navegación entre módulos funciona.
- La documentación deja claro qué es mock y qué queda pendiente.
