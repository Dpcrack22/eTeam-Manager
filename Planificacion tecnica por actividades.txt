eTeam Manager — Guía técnica por actividades

Fecha: 11 de marzo de 2026

======================================================================

OBJETIVO DEL DOCUMENTO

Este documento sirve como anexo práctico de la planificación extendida.
Su función es indicar, para cada actividad de cada sprint:

- qué archivos conviene tocar
- qué archivos nuevos conviene crear
- qué responsabilidad debería tener cada archivo
- qué decisiones ayudan a que el proyecto no se vuelva caótico

No es una regla cerrada, pero sí una guía recomendada para que el desarrollo
sea más ordenado y escalable.

REGLAS PRÁCTICAS PARA TODO EL PROYECTO

1) No meter toda la lógica en js/main.js
   main.js debe quedarse para inicialización general y utilidades globales.

2) No duplicar HTML de cabecera o menú en cada página
   Para eso están includes/header.php, includes/sidebar.php y footer.php.

3) No hacer un CSS distinto por cada página si no hace falta
   main.css debe llevar la mayor parte del sistema visual.

4) Separar datos mock de la interfaz
   Los mocks deben vivir aparte de los scripts que pintan la UI.

5) Cada página debe tener una responsabilidad clara
   Ejemplo: teams.php lista equipos, team-detail.php muestra uno concreto,
   team-form.php crea o edita.

SPRINT 1 — BASE VISUAL, LAYOUT Y AUTH

ACTIVIDAD 1 — Preparar estructura general de la aplicación

Archivos a tocar:
- index.php
- css/main.css
- js/main.js

Archivos recomendados para crear:
- includes/head.php
- includes/header.php
- includes/sidebar.php
- includes/footer.php
- includes/layout-start.php
- includes/layout-end.php

Para qué sirven:
- head.php: metadatos, fuentes y carga de CSS.
- header.php: barra superior compartida.
- sidebar.php: navegación principal.
- footer.php: pie común.
- layout-start.php y layout-end.php: apertura y cierre del layout general.

Recomendación:
Antes de hacer módulos, cerrad una plantilla base reutilizable. Si no lo hacéis
ahora, luego tendréis que reescribir todas las páginas.

ACTIVIDAD 2 — Diseñar layout compartido y navegación principal

Archivos a tocar:
- includes/header.php
- includes/sidebar.php
- css/main.css

Archivos recomendados para crear:
- pages/dashboard.php
- pages/teams.php
- pages/calendar.php
- pages/scrims.php
- pages/boards.php
- pages/notes.php
- pages/settings.php

Para qué sirven:
- Cada archivo de pages representa una vista navegable del sistema.

Recomendación:
Aunque al principio estén casi vacíos, cread ya los archivos de las páginas.
Eso os permite montar la navegación real desde el sprint 1.

ACTIVIDAD 3 — Consolidar identidad visual global

Archivos a tocar:
- css/main.css
- ui-kit.html

Archivos recomendados para crear:
- includes/ui-helpers.php

Para qué sirve:
- ui-helpers.php puede servir más adelante para helpers sencillos de clases,
  títulos o migas de pan si veis que repetís mucho bloque visual.

Recomendación:
Usad ui-kit.html como referencia visual, pero haced que la verdad final viva en
main.css y en las páginas PHP reales.

ACTIVIDAD 4 — Construir login y registro

Archivos a tocar:
- index.php
- js/main.js

Archivos recomendados para crear:
- pages/login.php
- pages/register.php
- js/auth.js

Para qué sirven:
- login.php y register.php: versión separada de acceso si más adelante dejáis de
  usar modal.
- auth.js: validación e interacciones de autenticación.

Recomendación:
Aunque ahora tengáis modal en la landing, os conviene dejar prevista una ruta de
login independiente. Luego será mucho más fácil proteger páginas.

ACTIVIDAD 5 — Validaciones front y sesión simulada

Archivos a tocar:
- js/main.js
- js/auth.js

Archivos recomendados para crear:
- js/session.js

Para qué sirve:
- session.js debe tener funciones tipo loginDemo, logoutDemo, isAuthenticated,
  getCurrentUser y requireAuth.

Recomendación:
Separar sesión de auth os ahorra mezclar validación de formularios con control
de acceso.

RECOMENDACIONES EXTRA QUE OS VAN A AHORRAR PROBLEMAS

1) Cread primero el esqueleto completo de páginas vacías
   Tener ya dashboard.php, teams.php, calendar.php, scrims.php y demás os ayuda
   a probar navegación desde el principio.

2) Usad includes para todo lo repetido
   Header, sidebar, footer y carga de recursos no deberían repetirse.

3) Mantened los mocks cercanos a la estructura de la base de datos
   Si el mock ya se parece al esquema real, el salto a backend será más fácil.

4) No sobrecarguéis el sprint 1 con demasiada lógica
   Lo importante ahí es cerrar base visual, estructura y auth simulada.

5) Cada módulo debería tener como mínimo una página y un script propio
   Eso os da orden mental y técnico.

6) Revisad siempre qué depende de organización activa y qué depende de equipo activo
   Esa diferencia os va a afectar en dashboard, scrims, calendar, boards y notes.

7) Priorizad primero la navegación real y luego el detalle fino
   Es mejor una app completa simple que dos módulos muy bonitos y el resto roto.

======================================================================

CONCLUSIÓN

Si seguís este reparto de archivos, el proyecto va a quedar mucho más limpio,
más fácil de presentar y mucho mejor preparado para conectar backend después.

La clave es esta:

- includes para layout compartido
- pages para vistas
- js/mocks para datos simulados
- js/modules para lógica por dominio
- main.css y main.js solo para lo global

Con esa base, cada sprint se vuelve mucho más fácil de ejecutar.
